# Inventorted

## CS-360 Portfolio Reflection

**Briefly summarize the requirements and goals of the app you developed. What user needs was this app designed to address? What screens and features were necessary to support user needs and produce a user-centered UI for the app? How did your UI designs keep users in mind? Why were your designs successful?**

Inventorted is an inventory tracking app for small business owners who need to know what they have, how much is left, and when something runs out. It replaces spreadsheets and paper logs with a dedicated mobile tool that's fast enough to use in the middle of receiving a shipment or pulling stock. The app has a login screen that doubles as account creation, and an inventory screen that keeps everything in one place: add item fields at the top, a scrollable grid in the middle, and SMS notification controls at the bottom. I kept the add-item fields on the same screen as the grid because the target user is someone working quickly and doesn't want extra navigation. Tapping a row opens a dialog to update the quantity rather than navigating to a detail screen, which keeps the interaction tight. The grid has column headers so the data is immediately readable, and there's an empty state message when no items exist yet so the screen doesn't just look broken. Those decisions came from thinking about the warehouse worker who's doing this with one hand while carrying something. The SMS alert system sends a notification when an item hits zero stock, and the user can toggle it on and off without revoking the system permission. If they deny the permission entirely, everything else still works.

**How did you approach the process of coding your app? What techniques or strategies did you use? How could those techniques or strategies be applied in the future? How did you test to ensure your code was functional? Why is this process important, and what did it reveal? Where did you have to innovate to overcome a challenge? In what specific component of your mobile app were you particularly successful in demonstrating your knowledge, skills, and experience?**

I built the UI layouts in XML first during Project Two, then added the functional Java code on top in Project Three. I started with the database helper class since everything else depends on it, then wired up login, then CRUD, then SMS. I used Logcat to debug issues, which turned out to be critical when the add button wasn't working and I needed to trace exactly where the flow was stopping. The biggest technical challenge was a ScrollView/RecyclerView nesting issue where the grid wasn't rendering items even though the data was being inserted correctly. The logic was fine but the view wasn't displaying, and the fix was swapping ScrollView for NestedScrollView, which handles the child measurement properly. That's the kind of thing where the code looks right but the framework has a quirk you have to work through. Testing was all done through the Android Emulator, running through each flow manually: wrong credentials, adding items, editing quantities, deleting rows, granting and denying SMS permission, and confirming the zero-stock alert fires. The component I'm most confident in is the database layer. The DatabaseHelper class handles both tables, exposes methods for every operation the app needs, and the flow from button press to database write to UI refresh works reliably. Getting that right made everything else straightforward since the login, the grid, and the SMS alerts all just call into the same helper class.
